# ISO/IEC/IEEE 29148

> Baseado na ISO/IEC/IEEE 29148:2018 — Systems and software engineering — Life cycle processes — Requirements engineering.

## O que é

Norma internacional que define os processos e os produtos de trabalho da **engenharia de requisitos** ao longo do ciclo de vida de sistemas e software, incluindo como especificar, analisar, verificar e validar requisitos.

## Processos de requisitos cobertos

- **Stakeholder Needs and Requirements Definition** — elicitação e definição das necessidades e requisitos das partes interessadas, na perspectiva de uso do sistema (independente de solução).
- **System/Software Requirements Definition** — tradução dos requisitos de stakeholders em requisitos técnicos do sistema/software a ser construído.

## Características de um requisito individual bem escrito

A norma define características que cada requisito, isoladamente, deve satisfazer:

- **Necessário** — descreve uma capacidade, característica ou restrição essencial; sua remoção causaria uma deficiência.
- **Apropriado (nível de abstração correto)** — o nível de detalhe é adequado ao nível da entidade que o requisito descreve.
- **Não ambíguo** — tem apenas uma interpretação possível.
- **Completo** — não precisa de material adicional (diagramas, tabelas) para ser entendido; não há "TBD" (a definir).
- **Singular** — expressa uma única capacidade/característica/restrição (uma única declaração "shall").
- **Factível (feasible)** — pode ser atendido dentro das restrições de custo, prazo e tecnologia.
- **Verificável** — existe um método finito e de custo razoável para confirmar que o requisito foi atendido.
- **Correto** — representa fielmente a necessidade da qual se origina.
- **Conforme** — segue o padrão de estilo definido (template de sentença) para uso consistente de termos.

## Características de um conjunto de requisitos

Além das características individuais, um **conjunto** de requisitos deve ser:

- **Completo** — nenhum requisito necessário está faltando.
- **Consistente** — nenhum requisito conflita com outro.
- **Factível (affordable)** — o conjunto pode ser atendido dentro das restrições do projeto.
- **Delimitado (bounded)** — mantém a fronteira/escopo pretendido.

## Estrutura de sentença recomendada

A norma recomenda um padrão de sentença para requisitos, tipicamente da forma:

> "O `<sujeito/sistema>` **shall** (deve) `<ação/capacidade>` `<sob que condição>` `<critério de desempenho, se aplicável>`."

Usar sempre um verbo modal consistente (`shall` para obrigatório) e evitar termos vagos ("deve ser fácil de usar", "deve ser rápido" sem critério mensurável).

## Tipos de requisitos

Funcionais, de desempenho, de interface, de dados, de segurança, de confiabilidade, de usabilidade e outros atributos de qualidade (não funcionais) — cada um documentado com o mesmo rigor de característica individual descrito acima.

## Relevância para este agente

Os critérios de "requisito bem escrito" desta norma (não ambíguo, singular, verificável, conforme) são a base para o agente validar critérios de aceitação e regras de negócio gerados, complementando os critérios INVEST (ver [[invest]]), que se aplicam à User Story como um todo.
